# L1: Active Context

> L1 is the top tier of the Semantic Forgetfulness memory hierarchy. It is the context window — what the LLM actually attends over on every forward pass. Content in L1 is at full semantic fidelity: either raw input tokens or fully enriched reconstructions from the L2→L1 enrichment pass. It is the only tier the model directly reads.

---

## What L1 Stores

Each L1 entry corresponds to one context segment (≈20 tokens, sentence-boundary aligned), held as a full token sequence:

```
L1Entry {
  id:                    UUID
  tokens:                Tensor[T]        # full token sequence (raw or reconstructed)
  metadata: {
    importance_score:    float            # attention concentration score at encoding time
    last_accessed:       timestamp
    source_position:     int              # original token position in conversation
    session_id:          str
    is_reconstructed:    bool             # false = raw input; true = promoted from L2
  }
}
```

Two content origins populate L1:
- **Raw input:** new tokens from the current conversation (user turns, assistant output, tool results) enter L1 directly as they arrive
- **L2 promotion:** a mid-compression CE tensor in L2 is passed through the reconstructor's L2→L1 query-conditioned enrichment pass, which injects the CE directly into L1 as soft tokens

Both are token sequences. The model does not distinguish between them during attention — they occupy the same context window. The `is_reconstructed` field exists for the cache controller's bookkeeping and for training signal analysis.

---

## Storage Architecture

**L1 lives in GPU VRAM. It is the context window.**

L1 is not a separate data store — it is the token buffer that gets passed to the LLM's forward pass on every turn. The cache controller maintains a parallel metadata index over L1 entries (importance scores, access times, segment IDs) that is stored in CPU memory. The metadata index is what enables the eviction decision; the token content itself lives on GPU.

L1 capacity is hard-bounded by the base model's context window size (e.g., 8,192 tokens for a Llama 3.2-3B, 128k for larger models). Within that bound, the active set of segments is managed by the cache controller.

---

## Retrieval

**There is no retrieval step for L1.** The model attends over all L1 content on every forward pass. A query "hits" L1 if the segment it needs is present in the token buffer at query time. There is no lookup, no similarity search, no explicit fetch — presence is retrieval.

### Miss Detection
The cache controller detects an L1 miss by checking whether query-relevant segments are absent from the current token buffer. The operational definition (see P9.1) is:

> A segment is "needed" by a query if the cosine similarity between the query's MiniLM embedding and the segment's MiniLM fingerprint (stored in the metadata index) exceeds threshold θ.

If a needed segment is absent from L1, the miss is classified:
- **Soft miss:** the segment is present in L2 → trigger L2 retrieval + L2→L1 enrichment pass
- **Hard miss:** the segment is absent from both L1 and L2 → trigger L3 retrieval + full reconstruction pipeline

---

## Eviction

When L1 is full and new content needs to enter, the cache controller evicts the segment with the lowest eviction score:

```
eviction_score(s) = 0.6 * importance_score(s) + 0.4 * exp(-λ * (t_now - t_last_accessed(s)))
evict argmin_s eviction_score(s)
```

Evicted L1 entries are **compressed by the compressor to a mid-compression CE tensor** (`C_L2` slots) and written to L2 as an L2Entry with `origin = "l1_demotion"`. The token content is removed from the GPU context buffer. The segment remains in the entity graph.

**Eviction fires on content entry, not on a schedule.** When a new segment arrives (either raw input or an L2 promotion), if L1 is at capacity, exactly one eviction occurs before the new segment is inserted. There is no batch eviction or periodic sweep.

**The inverted cost model applies here:** unlike physical memory, keeping tokens in L1 costs quadratic attention compute on every subsequent forward pass. This means the eviction threshold should be calibrated to evict aggressively — tolerating higher L2 fault rates is often cheaper than retaining low-importance tokens in the attention window.

---

## What L1 Is Not

- **Not a CE store** — concept embeddings live in L2 and L3. L1 contains only token sequences at full fidelity, including soft tokens injected from L2 CEs.
- **Not a mid-compression store** — CE tensors live in L2 and L3. Content is not present in L1 until the query-conditioned enrichment pass has run and the CE has been injected as soft tokens.
- **Not searched** — there is no ANN index, entity graph, or similarity search over L1. Presence is the access mechanism.
- **Not persistent** — L1 resets entirely between sessions. It is a live inference buffer, not a memory store.

---

## Key Design Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Representation | Full token sequences (raw or reconstructed) | LLM attends over tokens; no intermediate format survives to L1 |
| Storage location | GPU VRAM (context buffer) | Required for forward pass; metadata index separately in CPU memory |
| Retrieval | None — presence is access | Explicit retrieval would require a second forward pass; direct attention is cheaper |
| Miss detection | Cosine similarity between query embedding and segment fingerprints | Operationalizes "needed" without requiring ground truth labels |
| Eviction trigger | On new content entry when L1 is full | Eviction is synchronous with admission; no background sweep needed |
| Eviction target | L2 (compress to mid-compression CE tensor) | Preserves content in retrievable form one tier down |
| Eviction policy | Importance-weighted LRU | Aligns with training objective; importance score trained by cache-miss signal |
| Promotion prerequisite | L2→L1 enrichment pass required before promotion | Guarantees L1 holds query-conditioned CE soft tokens — not raw mid-compression CEs |
