# L2: Semantic Working Set

> L2 is the middle tier of the Semantic Forgetfulness memory hierarchy. It holds mid-compression CE tensors — concept embeddings produced either from L1 demotions or L3 promotions — and serves as both a retrieval target for L1 misses and a grounding corpus for the reconstructor. It is the tier that makes reconstruction trustworthy.

---

## What L2 Stores

Each L2 entry corresponds to one context segment (≈20 tokens, sentence-boundary aligned), held as a mid-compression CE tensor:

```
L2Entry {
  id:                    UUID
  ce_tensor:             Tensor[C_L2, d]  # mid-compression CE in LLM token embedding space
  semantic_fingerprint:  Tensor[768]      # frozen MiniLM embedding, computed at L1 admission
  confidence_scores:     [(slot, float)]  # per-slot confidence from reconstructor (if promoted from L3)
                                          # null if demoted directly from L1
  metadata: {
    importance_score:    float            # attention concentration score, set at original encoding time
    last_accessed:       timestamp
    source_position:     int              # original token position in conversation
    session_id:          str
    origin:              "l1_demotion" | "l3_promotion"
    grounding_used:      bool             # if l3_promotion: was L2 neighbor grounding used?
  }
}
```

Sanity anchors (boundary sentences, entity list) are not stored in L2Entry — they live in the cache controller's metadata index and are passed to the compressor and reconstructor at call time.

Two content origins produce L2 entries:
- **L1 demotion:** a token segment evicted from L1 is compressed by the compressor to a mid-compression CE tensor (`C_L2` slots) and written to L2
- **L3 promotion:** a high-compression CE tensor retrieved from L3 is passed through the reconstructor's L3→L2 fidelity pass, producing a mid-compression CE tensor that is stored in L2 before further enrichment

Both arrive as CE tensors. The `origin` field distinguishes them for debugging and training analysis, but they are treated identically by the retrieval and eviction machinery.

---

## Storage Architecture

**L2 lives in CPU DRAM. It does not occupy GPU VRAM.**

L2 entries are CE tensors — more compact than L1's full token sequences and of fixed size per segment (`C_L2 × d` floats). The full L2 working set for a session fits comfortably in DRAM without any disk interaction.

One structure:

### Entity Graph (Shared with L3)
The in-memory entity co-occurrence graph spans **both L2 and L3 entries**. Nodes are named entities, dates, and key noun phrases sourced from `sanity_anchors.entities` (extracted at L1 admission). Edges encode co-occurrence within the same segment or embedding cosine similarity above threshold τ.

L2 entries are registered into this graph at write time using their sanity anchor entity list. This is the primary lookup mechanism for L2: given a query or a segment being reconstructed, the entity graph surfaces L2 neighbor CE tensors in <1ms via 1-hop BFS. These neighbor CEs are the grounding corpus for the reconstructor's context-grounded expansion.

---

## Retrieval

### Trigger: L1 Cache Miss
L2 is queried only when content needed by the current query is absent from L1. L2 is not scanned proactively on every turn.

### Retrieval Pipeline

```
1. L1 miss detected by cache controller
       ↓
2. Extract entities / key terms from current query
       ↓
3. Entity graph lookup → candidate L2 entries (1-hop BFS)    <1ms
       ↓
4. Score candidates by semantic similarity to query           ~3ms
   (cosine between query MiniLM embedding and L2 entry fingerprints)
       ↓
5. Select top-k segments
       ↓
6. Pass CE tensors to reconstructor L2→L1 enrichment pass
       ↓
7. Inject enriched CE directly into L1 as soft tokens
       ↓
8. Log access for training signal
```

If the entity graph returns no candidates (sparse query, or L2 is cold), fall back to linear scan over L2 fingerprints — acceptable at L2 scale (~hundreds of entries).

---

## Warm-Start Protocol

L3 demotion is implicitly gated by `l2_capacity`. L2→L3 eviction cannot fire until L2 is full — meaning the first `l2_capacity` segments evicted from L1 always land in L2, naturally seeding it before any L3 activity begins. There is no separate warm-start hyperparameter.

**Why this matters for L2:** The reconstructor's context-grounded expansion relies on L2 neighbor CEs being present. If L2 is empty or sparse when the first L3 retrieval is attempted, grounding fails and the reconstructor falls back to sanity anchors only. The warm-start ensures L2 is seeded with real, high-quality CE tensors — compressed from the model's own active context — before L3 demotion begins.

---

## Eviction

When L2 capacity is exceeded, entries are evicted to L3:

```
eviction_score = 0.6 * importance_score + 0.4 * exp(-λ * (t_now - t_last_accessed))
evict argmin_s eviction_score(s)
```

Evicted L2 entries are **compressed to high-compression CEs by the compressor** (`C_L3` slots) and written to L3 as a new L3Entry. The entry is removed from the entity graph at L2 scope; its node remains in the graph if it is still referenced by an L3 entry.

---

## What L2 Is Not

- **Not raw tokens** — L2 holds mid-compression CE tensors, not full token sequences. L1 is the only tier with full fidelity token content.
- **Not text** — L2 holds CE tensors in LLM embedding space. No text is stored or generated at L2.
- **Not a query cache** — L2 is not searched on every turn, only on L1 misses. Frequent queries do not touch L2 if the content they need is already in L1.
- **Not the grounding source for L3 retrieval** — L2 CE tensors are the grounding source *for the reconstructor* during L3→L2 and L2→L1 passes. The HNSW ANN index and primary retrieval logic live in L3.

---

## Key Design Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Representation | Mid-compression CE tensor (`C_L2` slots) | Same embedding space as L1 and L3; no decode step needed; injectable directly as soft tokens |
| Storage location | CPU DRAM | CE tensors are fixed-size and compact; no GPU residency required until L2→L1 promotion |
| Retrieval mechanism | Entity graph 1-hop BFS + fingerprint rerank | Fast (<5ms total); entity nodes populated from sanity_anchors at write time |
| Graph scope | Shared with L3 | Enables reconstructor to ground L3 promotions in real L2 CE tensors without a separate index |
| Eviction target | L3 (compress to high-compression CEs) | Preserves content in retrievable form; eviction is lossy but recoverable |
| Warm-start gate | Implicit via `l2_capacity` | L2 cannot evict to L3 until full; no separate gate needed |
| Origin tracking | `l1_demotion` vs `l3_promotion` field | Enables offline analysis of reconstruction quality vs. original compression quality |
