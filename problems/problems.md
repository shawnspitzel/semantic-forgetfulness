> Organized by subsystem. Answers to these questions become the implementation spec.

## 0. Foundational: Mapping the Abstraction to Transformers

These questions must be answered first. Everything else depends on them.

**P0.1 — What exactly is L1, in terms of transformer internals?**
The most natural reading is that L1 = the KV cache (what the model actually attends over). But the KV cache is opaque by default. Options:
- L1 is a **subset of KV cache entries** we actively keep uncompressed
- L1 is a **separate attention stream** (cross-attention to a curated memory buffer)
- L1 is literally just **context window tokens** with no modification to the transformer at all
This choice determines whether this is a wrapper or a new architecture.

**P0.2 — Does this require modifying the base LLM?**
Can Semantic Forgetfulness be implemented as an external wrapper around a frozen LLM (inject retrieved content into the context), or does it require architectural changes (new attention heads, modified KV management, training new weights)? The wrapper approach is faster to prototype but may limit the model's ability to learn the importance function end-to-end. The goal is to come up with some light-weight, flexible interface with a pre-existing LLM, but our heavy modifications to inference, deployment, and continuous learning may make this difficult.

**P0.3 — What is the unit of compression?**
The compressor needs a defined chunk to operate on. Options:
- **Tokens** (finest granularity, most flexible, most expensive)
- **Fixed-size segments** (simpler, loses semantic coherence at boundaries)
- **Semantic chunks** (sentences, paragraphs — requires a chunker, but more natural)
- **Attention-head groups** (operating in KV space directly)
The answer affects every other component's input/output format.

**P0.4 — How is L3 different from RAG?**
L3 is described as "embedding index + similarity search." This is mechanically identical to RAG. The claimed difference is that the content in L3 was *generated by the model* (gists) rather than retrieved from an external corpus. Is this distinction sufficient to claim novelty over RAG + context compression? Need to articulate the precise difference and why it matters for the paper's framing.

## 1. Importance Signal

**P1.1 — What signal quantifies importance at encoding time?**
Several candidates are mentioned in the overview but none committed to:
- **Surprisal** (negative log probability): available at inference, cheap if tokens are being generated anyway. Problem: high surprisal = unexpected, not necessarily important.
- **Attention entropy / weight concentration**: tokens that receive concentrated attention from many heads may be "important." Requires inspecting attention weights.
- **Gradient saliency**: requires backprop through the model. Not feasible at inference without a separate pass.
- **Learned importance head**: a small auxiliary network predicts importance from hidden states. Requires training data.
Which signal is used determines what training data we need and whether importance estimation is feasible at inference time.

**P1.2 — Is surprisal a reliable importance proxy?**
Explicitly flagged in the overview as an open question. High surprisal at encoding time could indicate novelty (worth remembering) OR noise (worth forgetting). Low surprisal could indicate predictable filler OR core established context. Need an empirical test: does surprisal correlate with whether content is needed later?

**P1.3 — How is "task relevance" computed for active suppression?**
Active suppression requires knowing what is *counterproductive to the current task*. This requires a representation of the current task/goal. Options:
- The current query/prompt as a task representation (cosine similarity to content)
- A learned task-state embedding maintained throughout the session
- Rule-based: suppress content explicitly contradicted by recent tokens
This is the hardest importance question because it's inherently dynamic and query-dependent.

## 2. Compressor

**P2.1 — What architecture does the compressor use?**
The compressor takes a chunk of content (tokens, KV vectors, or embeddings) and produces a compressed gist. Options:
- **Cross-attention encoder**: attends over the segment, pools to a fixed-dim vector
- **Autoencoder** trained on reconstruction loss
- **Small LLM** that summarizes in natural language (token sequence, not vector)
- **Linear projection** of mean-pooled hidden states (simplest possible)
The output format of the gist must be decided here, since the reconstructor's input type depends on it.

**P2.2 — What is the gist format?**
- **Dense embedding vector**: compact, fast for similarity search, but requires a learned decoder for reconstruction
- **Short token sequence** (natural language summary): interpretable, can be fed directly to the LLM, but larger and noisier
- **Sparse vector**: interpretable, compressible, but less expressive
This choice also determines how L3 search works (vector index vs. token-based retrieval).

**P2.3 — What is the compressor's training objective?**
Several possible objectives conflict:
- Minimize reconstruction loss (reproduce original content) → conservative, low compression
- Maximize downstream task performance (does the gist help answer future queries?) → requires task labels
- Maximize L1 hit rate (indirect, sparse, delayed signal) → the stated primary objective, but hard to backprop through
- Teacher-forcing from a full-context model (distillation) → most tractable immediately
How do we combine these? Is the L1 hit rate objective sufficient on its own, or do we need an auxiliary reconstruction loss to stabilize early training?

**P2.4 — Does the compressor need to be trained jointly with the base LLM, or can it be trained independently?**
Joint training = the LLM learns to use compressed representations + the compressor learns what matters for the LLM. Independent training = simpler, faster, but the compressor doesn't know how the LLM will consume its output. Distillation approach (overview §4) implies independent training — is that good enough?

## 3. Reconstructor

**P3.1 — What architecture does the reconstructor use?**
The reconstructor takes a gist (from L3) and optionally surrounding context/source anchors, and produces a representation that can be promoted to L2/L1. Options:
- **Small generative LLM** that expands the gist into a sequence
- **Decoder head** trained to map gist embeddings back into the LLM's embedding space
- **Artifact reconstruction** rather than reconstructing tokens -> text, we reconstruct artifacts, kv facts, entity relationships, or summary embeddings, and then generate new text based off this recovered context with some LLM.
- **Direct injection** of the gist embedding into cross-attention (no reconstruction, just use the compressed form)
The last option reframes reconstruction as "use gist as-is" rather than "recover original" — potentially much simpler.

**P3.2 — How does the reconstructor produce uncertainty estimates?**
The overview specifies that reconstructed content must be *uncertainty-flagged* before L1 promotion. This is critical for preventing hallucination propagation. Options:
- **Bayesian / MC-dropout**: multiple stochastic forward passes, measure variance
- **Conformal prediction**: calibrated uncertainty from held-out reconstruction error
- **Auxiliary confidence head**: trained to predict reconstruction quality
- **Semantic similarity to sanity anchors**: if reconstructed content is far from a stored structural anchor, flag it
What is the threshold for "too uncertain to promote"? What happens to content that fails — does it stay in L3, get re-retrieved, or get dropped?

**P3.3 — What are "sanity anchors" concretely?**
The overview mentions storing "structural sanity anchors alongside gists." What exactly are these? Candidates:
- First/last sentence of the original segment (boundary markers)
- Named entities, numbers, dates (factual anchors)
- Sentence-level embeddings from a frozen encoder (semantic fingerprint)
- Key-phrase extraction (lightweight NLP)
The anchor must be small enough not to undermine the compression ratio, but rich enough to validate the reconstruction.

**P3.4 — What is the acceptable latency budget for L3 retrieval + reconstruction?**
This determines the practical feasibility of the L3 miss path. If reconstruction takes 500ms, that's a hard UX problem for interactive applications. Need a target latency and a measurement plan.

## 4. Cache Management & Promotion Policy

**P4.1 — How are cache sizes set and bounded?**
L1 = context window (bounded by model architecture). L2 and L3 size: fixed per session? Per user? Per domain? Dynamic? The overview notes cache size as a hyperparameter but doesn't specify the range or what governs the bound.

**P4.2 — What is the exact eviction policy?**
When L1/L2 is full and new content needs to be admitted, what gets evicted? Options:
- **LRU (Least Recently Used)**: simple, ignores importance
- **Importance-weighted LRU**: importance * recency score, evict lowest
- **Explicit importance threshold**: evict anything below threshold X
- **LFU (Least Frequently Used)**: based on access counts
This is the policy that the training signal directly optimizes — it needs to be fixed before training can be designed.

**P4.3 — What triggers promotion from L3 → L2 → L1?**
The overview mentions this as a hyperparameter (promote after N cache misses?) but doesn't define the mechanism. Questions:
- Does a single L3 hit trigger promotion, or only repeated hits?
- Is promotion immediate or deferred (promote at end of session)?
- Do we demote back (L1 → L2) if content stops being accessed? What's the demotion threshold?

**P4.4 — How is the importance score updated over time within a session?**
The importance of a piece of content can change as the conversation evolves. Initial encoding importance vs. retrospective importance. Does the system re-score content as new context arrives, or is the importance fixed at encoding time?

## 5. Training

**P5.1 — How is the cache-miss signal converted into a gradient?**
This is the core training mechanism and it's underspecified. A cache miss is a discrete event — it can't be backpropagated through directly. Options:
- **Offline RL**: log (state, action, reward) tuples during deployment, train a policy offline
- **REINFORCE / policy gradient**: treat allocation decisions as a stochastic policy
- **Behavioral cloning from an oracle**: an oracle with full context labels which content is important in hindsight; student learns to predict this
- **Contrastive loss**: pairs of (important, unimportant) content at encoding time
Which framing determines the training infrastructure, data collection strategy, and stability of learning.

**P5.2 — Where does learning live? (Explicitly flagged as unresolved in overview)**
Two options:
- **Weight updates at inference**: the compressor/importance model is updated from cache-miss signals during deployment. Risk: catastrophic forgetting, instability, privacy (weights encode session data).
- **Cache state updates only**: no weight changes at inference; only the cache contents change. Learning is offline, periodic, safe. Risk: adaptation is slow, session-specific patterns not leveraged.
A hybrid (frequent lightweight cache updates + infrequent offline weight updates) may be optimal. Decision needed before any training code is written.

**P5.3 — What data is needed for distillation (the fastest initial path)?**
Distillation from a teacher model requires:
- A teacher with access to full context (what model? GPT-4o? A large open-weights model?)
- Pairs of (full-context output, compressed-context input) for training the student
- A task distribution to train on (general? Long-document specific?)
How large a dataset? What's the minimum to see a useful compression signal?

**P5.4 — How do we handle catastrophic forgetting if weight updates happen at inference?**
If the importance model is updated from session data, it may overfit to recent sessions and forget how to handle earlier patterns. Mitigations: elastic weight consolidation (EWC), experience replay, low-rank adaptation (LoRA) updates only. Which approach is compatible with the latency budget?

## 6. Suppression (Active Forgetting)

**P6.1 — Is suppression a separate module or does it emerge from the compressor?**
The overview explicitly asks this question. Arguments for separate:
- Different objective (counterproductive vs. unimportant)
- Different signal (task-relevance vs. general importance)
Arguments for unified:
- Simpler architecture
- Shared representations may transfer
- Two separate modules double the training complexity
Proposed resolution: start with compressor-only, add suppression as an ablation once the core system works.

**P6.2 — What is the ground truth for "counterproductive" content?**
To train a suppression mechanism, we need examples of content that hurt performance when present. How do we construct this? Options:
- Inject adversarial/conflicting context and measure performance degradation
- Use datasets with known misleading passages (e.g., debunking tasks, contradictory documents)
- Retrospective labeling: content that, when removed from context, *improves* model output
This is the hardest training data problem in the whole system.

**P6.3 — Does suppression interact with L3 promotion?**
If content is actively suppressed (demoted because it's counterproductive), should it be promotable back if the task changes? Or is suppression permanent for the session? The brain analogy suggests temporary suppression (retrieval inhibition releases under different cues), but implementing session-scoped temporary suppression adds significant state management complexity.

## 7. Cross-Session Memory

**P7.1 — Is L3 persistent across sessions?**
The overview flags this as "a different engineering problem." Decision needed: is out-of-scope for the initial research contribution, or is cross-session L3 a core feature? If in scope, what storage backend? If out of scope, state it explicitly so the paper can scope correctly.

**P7.2 — If L3 persists, how is user privacy handled?**
L3 gists are derived from user conversation content. Storing them raises data governance questions. For a research prototype, this may not matter, but it affects deployment framing.

## 8. Interaction with Existing Systems

**P8.1 — How does this interact with RAG pipelines?**
The overview notes potential overlap or synergy. Concretely: if a system already has RAG, does L3 replace the external corpus, sit alongside it, or does RAG become the retrieval mechanism *for* L3? This could be a strength (L3 = personalized RAG built from the model's own gists) or a source of confusion.

**P8.2 — Does the system require a specific model architecture?**
Ideally the system works across model families (LLaMA, Mistral, GPT-style). But if L1 is defined as "a subset of KV cache entries," access to the KV cache is required, which varies by model and inference framework. Which frameworks expose the KV cache in a way that supports this? (vLLM, HuggingFace `generate()` with `past_key_values`, etc.)

**P8.3 - How will we integrate our system into an agentic system? How will we test this?**
- If we want to support agentic workflows, how can we integrate our system into agentic models and infrastructure? What is our use case for this model? Coding? Conversation? What makes the most sense given our tradeoffs?

## 9. Evaluation & Measurement

**P9.1 — How is L1 hit rate defined and measured rigorously?**
The L1 hit rate is the central training objective. But operationally, when is a query considered an "L1 hit"? Options:
- The relevant information is present in L1 at query time (requires knowing what's "relevant")
- The model produces correct output without needing L2/L3 retrieval (task-based proxy)
- Explicit tracking: log which cache level each context segment was retrieved from
Need a formal definition before this can be used as a training signal.

**P9.2 — What is the baseline?**
To show improvement, we need a baseline that represents "no Semantic Forgetfulness." Options:
- Naive truncation (drop oldest tokens when context full)
- Sliding window attention
- Existing compression methods (CompLLM, FastKV, StreamingLLM)
- Full context (oracle upper bound)
The comparison to CompLLM is most important for the novelty claim.

**P9.3 — What is the minimum viable prototype to validate the core claim?**
The core claim is: "a model can learn to allocate context by importance, and cache-miss signals improve that allocation over time." What is the simplest experiment that could confirm or disconfirm this? This scopes the first implementation milestone.

## 10. Compression Ratios

**P10.1 — What compression ratios are feasible at each level?**
L1 → L2: moderate compression (how much?) without significant semantic loss.
L2 → L3: aggressive compression (how much?) with accepted loss.
The ratio determines memory capacity and reconstruction difficulty. Need empirical baselines: at what compression ratio does reconstruction fidelity collapse?

**P10.2 — At what context length does context rot justify this overhead?**
The overhead (compression, retrieval, reconstruction) is only worthwhile if context rot is hurting performance enough to compensate. Need an empirical threshold: below X tokens, standard attention is fine; above X, Semantic Forgetfulness helps. This determines the system's activation condition.
